# 为什么选择 Celer？

> *Celer 是一个面向企业场景的 C/C++ 包管理器，专注解决依赖管理里最贵、最痛、最容易反复踩坑的部分。*

你可能已经在用 **Conan**、**Vcpkg**、**XMake**。它们很强，但在企业项目里，团队真正被拖慢的往往不是“能不能下载到库”，而是下面这些高频问题：

## 🎯 Celer 重点解决的痛点

### 1. 📦 新库接入慢，不是因为代码难，而是流程重

**你遇到的痛：**

- 用现成库很快，但一旦引入“没现成配方”的库，接入周期会突然拉长
- 开发者被迫学习工具专有脚本、补齐构建细节、处理安装路径
- 每次接新库都像“重新发明一次流程”

**实际代价：**

- 功能开发被阻塞，排期不可控
- 接入过程强依赖少数熟手，团队扩展困难
- 配置质量不稳定，后续维护成本持续上涨

**Celer 的做法：**

只需在 TOML 里声明构建系统类型（CMake/Make/Meson 等）和必要选项，Celer 负责标准化接入流程。  
你关注“这个库要什么”，不是“这套工具链怎么驯服”。

### 2. 🏢 项目之间相互污染，隔离不彻底就会反复返工

**你遇到的痛：**

- 不同项目对同一库需要不同开关，甚至互斥配置
- 全局安装/共享目录模式下，一个项目的改动会影响另一个项目
- 私有库和公共库混用时，边界容易失控

**实际代价：**

- “昨天能编过，今天突然不行”的偶发问题增加
- 团队大量时间消耗在排查环境差异，而不是业务开发
- 版本升级变成高风险动作，大家不敢动

**Celer 的做法：**

按项目隔离依赖版本、构建选项和私有库定义。  
每个项目有自己的可重现依赖配置，不与其他项目交叉污染。

### 3. 🔗 多子工程依赖漂移，平台越大越难控

**你遇到的痛：**

- 平台工程里每个子项目各自维护依赖，配置分散
- 时间一长就出现版本漂移，“名义同平台，实际不同栈”
- 统一升级时需要逐仓对齐，人工核查成本高

**实际代价：**

- 同一功能在 A 子项目通过、在 B 子项目失败，联调阶段反复出现“只在某仓复现”的问题
- 每次版本升级都要逐个子项目回归验证，测试与构建时间被线性放大
- 发布前常因某个子项目依赖版本落后而临时补丁或回退，直接挤占发布窗口

**Celer 的做法：**

用单一 TOML 做集中依赖定义，并自动生成统一 `toolchain_file.cmake`，子项目直接继承。  
一次更新，全局同步，减少“人肉对齐”。

### 4. ⚡ 构建缓存不可控，时间浪费在重复编译

**你遇到的痛：**

- 多平台依赖常靠手工预编译和共享目录维护
- 轻微配置变化就触发整套重编，难以判断哪些结果可复用
- 用 Git 或压缩包存制品，体积膨胀、传输慢、维护粗放

**实际代价：**

- CI/CD 和本地构建时间持续走高
- 存储与带宽成本被动上涨
- 团队对“改动一次要等多久”缺乏预期

**Celer 的做法：**

基于环境、编译选项和依赖链生成哈希键，精确命中可复用制品。  
该复用就复用，该失效就失效，减少重复编译和人工清理。

### 5. 🔍 冲突发现太晚，问题总在集成阶段爆雷

**你遇到的痛：**

- 深层依赖冲突（尤其菱形依赖）很难在早期看出来
- 常在联调或运行阶段才暴露，定位链路长
- 手工排查依赖树，低效且容易漏

**实际代价：**

- 版本回退和临时补丁成为常态
- 关键节点返工，影响发布时间
- 团队对依赖升级天然保守

**Celer 的做法：**

在构建阶段做依赖版本一致性检查，并输出可操作的冲突信息。  
把“上线前爆雷”前移为“构建时可见”。

### 6. 🤝 跨公司协作门槛高，环境交付难复制

**你遇到的痛：**

- 对外协作常需要同步完整构建环境，落地复杂
- 合作方机器环境不一致，初始化成本高
- “你这边能编，我这边不行”反复出现

**实际代价：**

- 对接周期长，沟通成本高
- 问题排查依赖核心人员远程救火
- 交付质量受环境不可控因素影响

**Celer 的做法：**

通过可移植的 `toolchain_file.cmake` 交付依赖环境（相对路径、自包含）。  
合作方只需设置 `CMAKE_TOOLCHAIN_FILE`，即可快速获得一致构建上下文。

---

## 🚀 Celer 适合什么团队

- 需要频繁接入新 C/C++ 三方库以及内部开发公共基础库
- 管理多个平台/多个子工程的企业项目
- 对构建速度、可重现性和协作效率有硬要求
- 希望把依赖管理从“经验活”变成“工程化流程”
